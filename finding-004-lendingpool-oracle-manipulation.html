Security Audit Finding #004

Collateral Under-Pricing via Single-Pool
Spot Price Oracle Manipulation

LendingPool.sol Custom training example Educational Audit Exercise
Critical
The lending pool determines collateral value by reading a single DEX pool's live token reserves. A flash-loan-funded attacker can distort that pool's price within a single transaction, causing the protocol to massively over-value deposited collateral, then borrow far more than that collateral could legitimately support — leaving the protocol with unrecoverable bad debt.
Description

LendingPool allows users to deposit ETH as collateral and borrow USDC against it. To determine how much a user's ETH collateral is worth, the contract calls getPrice(), which reads the live token reserves of a single Uniswap-style ETH/USDC liquidity pool and derives price directly from that ratio.

This is a spot price — it reflects only what is currently sitting in one pool, at this exact moment, with no resistance to short-term manipulation. AMM pool ratios shift with every trade; a sufficiently large trade shifts the ratio sharply, even if only for the duration of a single transaction.

Because Solidity transactions are atomic and flash loans allow borrowing large sums with no upfront capital — repayable within the same transaction — an attacker can temporarily distort the pool's reported price, exploit that distortion against LendingPool, and repay the flash loan before the transaction ends, walking away with the difference as profit at the protocol's expense.

Vulnerable Code
LendingPool.sol — getPrice() and depositCollateral() (vulnerable)
contract LendingPool {
    IUniswapPool public immutable priceSource;
    mapping(address => uint256) public collateralValue;
    mapping(address => uint256) public borrowed;

    // ❌ Reads live reserves from a single pool — no time-weighting, no external check
    function getPrice() public view returns (uint256) {
        (uint256 ethReserve, uint256 usdcReserve) = priceSource.getReserves();
        return (usdcReserve * 1e18) / ethReserve; // spot price, manipulable within one tx
    }

    // ❌ Collateral value calculated using the manipulable spot price above
    function depositCollateral() external payable {
        uint256 price = getPrice();
        collateralValue[msg.sender] += msg.value * price / 1e18;
    }

    function borrow(uint256 _amount) external {
        require(borrowed[msg.sender] + _amount <= collateralValue[msg.sender] * 80 / 100,
            "Exceeds LTV");
        borrowed[msg.sender] += _amount;
        USDC.transfer(msg.sender, _amount);
    }
}
Impact

An attacker exploiting this flaw can, within a single transaction:

  • Artificially inflate the reported price of ETH by dumping a flash-loaned amount of USDC into the pool
  • Deposit a small amount of real ETH as collateral while the price is distorted, causing collateralValue to be recorded far above the ETH's real worth
  • Borrow USDC up to 80% of that inflated, false collateral value — far more than the real ETH deposited should support
  • Repay the flash loan and keep the excess borrowed USDC as profit
  • Leave LendingPool under-collateralized, with genuine bad debt that legitimate depositors ultimately absorb
Real-World Precedent — Multiple Flash Loan Oracle Attacks (2020–2022) This exact pattern — flash-loan-funded manipulation of a single DEX pool used as a price source — has been responsible for tens of millions of pounds in losses across DeFi lending protocols, including well-documented incidents such as the bZx (February 2020) and Cheese Bank (November 2020) exploits. In each case, the root cause was the same architectural flaw: trusting a manipulable, single-source spot price for high-stakes financial logic.
Proof of Concept

Executable entirely within one transaction; no persistent capital required beyond gas.

  1. Attacker takes out a large flash loan of USDC, repayable within the same transaction.
  2. Attacker swaps that USDC into the Uniswap ETH/USDC pool used by priceSource, sharply reducing the pool's ETH reserve relative to USDC and inflating the reported ETH price.
  3. Attacker calls:
    LendingPool.depositCollateral{value: 1 ether}()
    recording a collateralValue based on the now-inflated price.
  4. Attacker calls:
    LendingPool.borrow(inflatedAmount)
    withdrawing USDC far exceeding the real value of the 1 ETH deposited.
  5. Attacker swaps back through the pool (or a separate route) to restore its balance and repay the original flash loan.
  6. Transaction completes; attacker keeps the excess borrowed USDC as profit. LendingPool is left holding bad debt against the now-undervalued real collateral.
Cost to attacker: gas fees + flash loan fee (typically ~0.05–0.09%)  |  Time to exploit: one transaction
Recommendation

Never use a single pool's live spot price for high-stakes valuation logic. Two established mitigations:

Before — Vulnerable
function getPrice() public view returns (uint256) {
    (uint256 ethReserve, uint256 usdcReserve) = priceSource.getReserves();
    return (usdcReserve * 1e18) / ethReserve; // ❌ manipulable within one tx
}
After — Fixed (Chainlink price feed)
AggregatorV3Interface public immutable priceFeed;

// ✅ Chainlink aggregates across many sources over time — not manipulable in one tx
function getPrice() public view returns (uint256) {
    (, int256 price, , uint256 updatedAt, ) = priceFeed.latestRoundData();
    require(price > 0 && block.timestamp - updatedAt < 1 hours, "Stale or invalid price");
    return uint256(price);
}
Alternative — TWAP Where a Chainlink feed isn't available for a given asset pair, a Time-Weighted Average Price sampled over a sufficiently long window (e.g. 30+ minutes) is a reasonable alternative — a single-transaction flash loan cannot meaningfully shift an average calculated over that timeframe. Whichever approach is chosen, price data should always include a staleness check before being trusted for financial logic.
