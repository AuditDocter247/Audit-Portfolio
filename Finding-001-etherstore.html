Reentrancy: External Call Before State Update in withdraw()

Severity Justification
Rated Critical — an attacker can drain the contract's entire ETH balance in a single transaction with no privileged access required. The vulnerability is trivially exploitable, requires no special conditions, and results in total, irreversible loss of funds for all depositors. Historical exploits of this class have resulted in losses exceeding £100 million.
Description

The EtherStore contract allows users to deposit and withdraw ETH. The withdraw() function contains a classic reentrancy vulnerability: it sends ETH to the caller via a low-level .call{value: ...}() before setting the caller's balance to zero. This ordering is the root cause of the vulnerability.

Because the balance is not zeroed before the external call, any contract receiving the ETH can immediately call withdraw() again during that same transaction. At the moment of this recursive call, the victim contract's state still records the original balance — so the balance check passes, and more ETH is sent out. This loop continues until the contract is fully drained.

This is not a subtle or edge-case bug. It represents a failure to follow the Checks-Effects-Interactions (CEI) pattern — a fundamental Solidity best practice specifically designed to prevent this class of exploit. Any contract that sends ETH before updating internal accounting is potentially vulnerable to this attack.

Vulnerable Code
⚠ EtherStore.sol — withdraw() Solidity ^0.8.x 
 1function withdraw() public {
 2    uint bal = balances[msg.sender];
 3
 4    // CHECK — validates the caller has funds
 5    require(bal > 0);
 6
 6    // ❌ INTERACTION before EFFECT
 7    // ETH is sent to msg.sender HERE...
 8    (bool sent, ) = msg.sender.call{value: bal}("");
 9    require(sent, "Failed to send Ether");

10    // ❌ ...but balance is only zeroed AFTER
11    // An attacker re-enters before this line executes
12    balances[msg.sender] = 0;
13}

Lines 8–9 (highlighted) send ETH to an external address before line 12 zeros the balance. This ordering is the sole cause of the vulnerability.

Impact
  • Complete drainage of all ETH held by the contract in a single transaction, affecting every depositor simultaneously.
  • No privileged access is required — any address that has made even a minimal deposit can execute this attack.
  • Funds are unrecoverable once drained; there is no pause mechanism or rescue function in this contract.
  • The attack requires only one malicious contract and a small initial deposit; gas costs are the attacker's only barrier to entry.

Proof of Concept
  1. Attacker deploys a malicious contract (Attack.sol) containing a receive() function that calls EtherStore.withdraw() recursively.
  2. Attacker deposits a small amount of ETH (e.g., 1 ETH) into EtherStore via the malicious contract, establishing a legitimate non-zero balance.
  3. Attacker calls Attack.attack(), which triggers EtherStore.withdraw() for the first time.
  4. EtherStore checks the balance (passes), then sends 1 ETH to the malicious contract via .call.
  5. Before EtherStore can execute balances[msg.sender] = 0, the malicious contract's receive() fires and calls withdraw() again.
  6. The balance check passes again (balance is still 1 ETH in storage), so another 1 ETH is sent. Steps 4–6 repeat recursively.
  7. The loop continues until EtherStore's ETH balance is insufficient to send another payment, at which point the call stack unwinds.
  8. The attacker's contract holds all drained ETH. EtherStore's balance is now zero; all depositors have lost their funds.
Recommendation

Apply the Checks-Effects-Interactions (CEI) pattern: always update all internal state before making any external call. If the interaction reverts, the state change reverts with it — so no balances are double-spent.

Validate all conditions (require, revert)

✗ Before (Vulnerable) 
function withdraw() public {
  uint bal = balances[msg.sender];
  require(bal > 0);

  // Interaction first ❌
  (bool sent, ) = msg.sender
    .call{value: bal}("");
  require(sent);

  // Effect after ❌
  balances[msg.sender] = 0;
}
✓ After (Fixed — CEI) 
function withdraw() public {
  uint bal = balances[msg.sender];
  require(bal > 0);

  // Effect first ✅
  balances[msg.sender] = 0;

  // Interaction after ✅
  (bool sent, ) = msg.sender
    .call{value: bal}("");
  require(sent);
}

Additional mitigations to consider:

  • Add a reentrancy guard using a uint256 lock variable (OpenZeppelin's ReentrancyGuard pattern) as a defence-in-depth measure. Prefer uint256 over bool due to EVM storage reset costs post-EIP-3529.
  • For Solidity 0.8.24+, consider transient storage (EIP-1153) for the most gas-efficient reentrancy lock.
  • CEI alone is the primary fix; the guard is a belt-and-braces secondary control for complex codebases where CEI may be harder to audit visually.
