Security Audit Finding #002

Unrestricted Ownership Takeover via
Missing Access Control on setOwner()

AdminVault.sol Custom training example Educational Audit Exercise
Critical
Zero preconditions to exploit — any externally owned account can call setOwner() directly, assume full administrative control, and drain all held funds in a single transaction. Complete contract compromise. No special access, tools, or timing required.
Description

The AdminVault contract implements an owner-based access control pattern: only the designated owner address may call withdraw() and recover held funds. That restriction is correctly applied. However, the function responsible for updating the owner address — setOwner() — carries no corresponding restriction. It is declared public with no modifier, no require() guard, and no validation of msg.sender.

This creates a critical asymmetry: the contract protects the action (withdrawal) but leaves the gate to that action (ownership assignment) entirely unguarded. Any caller — including anonymous attackers — can invoke setOwner() with their own address as the argument, silently overwriting the legitimate owner and acquiring full administrative rights.

The vulnerability is a textbook example of privilege escalation through missing access control — one of the most consistently exploited classes in deployed smart contracts. It is directly analogous to a safe that locks its door but leaves the combination-change mechanism accessible to anyone who walks past.

Vulnerable Code
AdminVault.sol — setOwner() (vulnerable)
// ❌ No access control — any caller can invoke this function
// ❌ No require(msg.sender == owner) check
// ❌ No onlyOwner modifier
function setOwner(address _newOwner) public {
    owner = _newOwner; // ownership silently transferred to any caller
}
Impact

A successful exploit grants the attacker complete ownership of the contract. From that position, the attacker can:

  • Call withdraw() to drain the contract's entire ETH balance to their own address
  • Permanently lock the original deployer out of all owner-gated functions
  • In proxy-pattern contracts: upgrade logic, brick implementation, or redirect all funds
  • In token contracts: mint arbitrary supply, freeze transfers, or burn legitimate balances
  • Modify any critical protocol parameters now protected by the onlyOwner pattern
Real-World Precedent — Parity Wallet (November 2017) The Parity multi-sig wallet library contained an uninitialised initWallet() function — an analogous missing access control flaw. An attacker claimed ownership of the unprotected library contract, then accidentally triggered selfdestruct(), permanently freezing over £150M+ worth of ETH across hundreds of dependent multi-sig wallets. Funds remain inaccessible to this day. The root cause: a privileged state-changing function with no caller restriction.
Proof of Concept

No special tooling, flash loans, or privileged access required. Attack is executable in a single transaction.

  1. Attacker identifies the deployed AdminVault contract on-chain and inspects its public ABI or verified source code.
  2. Attacker observes that setOwner() is externally callable with no modifier or require() guard.
  3. Attacker calls:
    AdminVault.setOwner(attacker_address)
  4. The contract executes without reversion. The owner state variable is overwritten with the attacker's address.
  5. The original deployer is now permanently locked out. All require(msg.sender == owner) checks will fail for them.
  6. Attacker calls withdraw() — which now passes the ownership check — and transfers the contract's full ETH balance to their wallet.
Cost to attacker: gas fees only  |  Time to exploit: one block
Recommendation

Restrict setOwner() to the current owner only. Implement an onlyOwner modifier and apply it to all privileged state-changing functions. Additionally, guard against the zero address being set as owner — a mistake that would permanently and irreversibly brick access.

Before — Vulnerable
function setOwner(address _newOwner) public {
    // ❌ No access control
    owner = _newOwner;
}
After — Fixed
// ✅ Modifier restricts access to current owner only
modifier onlyOwner() {
    require(msg.sender == owner, "AdminVault: caller is not the owner");
    _;
}

// ✅ onlyOwner applied — ✅ zero-address guard added
function setOwner(address _newOwner) public onlyOwner {
    require(_newOwner != address(0), "AdminVault: new owner is zero address");
    owner = _newOwner;
}
Production Note Consider emitting an OwnershipTransferred event on every ownership change, enabling off-chain monitoring tools to alert on unexpected privilege transfers. For non-educational deployments, OpenZeppelin's Ownable contract provides a battle-tested, audited implementation of this exact pattern and is the recommended choice.
