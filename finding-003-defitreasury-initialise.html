Security Audit Finding #003

Unprotected Initialiser Front-Running Leading to
Ownership Hijack on initialise()

DeFiTreasury.sol Proxy Pattern Contract Custom training example Educational Audit Exercise
Critical
The contract uses a proxy-style initialiser instead of a constructor, but initialise() carries no restriction on who may call it. The first caller — legitimate deployer or attacker — becomes owner. An attacker monitoring the public mempool can front-run the deployer's own initialisation transaction, seize ownership, and drain all funds subsequently deposited into the treasury.
Description

DeFiTreasury follows a proxy-pattern architecture. Because proxy contracts cannot rely on a normal Solidity constructor (constructors don't run in the context of a proxy's storage), state such as the owner address is instead set via a separate initialise() function, intended to be called once, immediately after deployment.

The function does include a re-entrancy-style guard against being called twice — an initialised boolean flips to true on first use and blocks any subsequent call. However, it applies no restriction on who may be the one to make that first call. There is no require(msg.sender == deployer) check, no constructor-set deployer address to compare against, and no atomic deploy-and-initialise step.

This turns initialisation into a public race condition. Deployment and initialisation happen as two separate, sequential transactions — and both sit briefly in the public mempool before confirming. Anyone watching the mempool can see a freshly deployed, not-yet-initialised DeFiTreasury instance and submit their own initialise() call with a higher gas price to confirm first — regardless of whether they have ever interacted with the contract before.

Vulnerable Code
DeFiTreasury.sol — initialise() (vulnerable)
contract DeFiTreasury {
    address public owner;
    bool    public initialised;
    mapping(address => uint256) public balances;

    // ❌ Guards against re-initialisation only
    // ❌ No check on WHO is allowed to call this first
    // ❌ No constructor-pinned deployer address to compare against
    function initialise(address _owner) public {
        require(!initialised, "DeFiTreasury: already initialised");
        initialised = true;
        owner = _owner; // first caller wins — attacker can pass their own address
    }

    function withdraw(uint256 amount) public {
        require(msg.sender == owner, "DeFiTreasury: not owner");
        payable(msg.sender).transfer(amount);
    }
}
Impact

Whoever wins the initialisation race becomes the permanent owner. An attacker who wins can:

  • Set themselves as owner before the legitimate deployer's transaction confirms
  • Call withdraw() to drain any funds subsequently deposited by users who trust the contract address
  • Permanently lock the real deployer out — the guard against re-initialisation now works against the legitimate team
  • In real deployments, redirect upgrade logic or brick the proxy implementation entirely
  • Cause reputational and financial damage before the project has even launched publicly
Real-World Precedent — Audius Governance Takeover (July 2022) An attacker exploited a contract in the Audius protocol that had been deployed but not yet properly configured, using it to gain control over governance and mint an estimated 18 million AUDIO tokens — reported at the time as roughly £4.9M in value. As with DeFiTreasury, the root cause was a privileged setup function left reachable by an unintended caller during the gap between deployment and proper configuration.
Proof of Concept

No special privileges required — only a mempool-monitoring bot and a willingness to pay a higher gas fee than the deployer.

  1. Deployer deploys the DeFiTreasury proxy. At this point owner is unset and initialised is false.
  2. Deployer broadcasts a follow-up transaction calling initialise(deployerAddress). It enters the public mempool and waits to be mined.
  3. Attacker's bot detects the pending, not-yet-confirmed deployment and spots the uninitialised contract.
  4. Attacker submits their own call with a higher gas price to be prioritised:
    DeFiTreasury.initialise(attacker_address)
  5. Attacker's transaction confirms first. initialised flips to true; owner is set to the attacker's address.
  6. The deployer's original transaction now reverts with "already initialised" — they are permanently locked out.
  7. Users, unaware anything is wrong, deposit funds into what they believe is the legitimate treasury.
  8. Attacker calls withdraw() — which now passes the ownership check — and drains the deposited funds.
Cost to attacker: gas fees only, plus a modest gas premium to win the race  |  Time to exploit: one block
Recommendation

Never leave a window where a deployed-but-uninitialised contract is publicly callable. Two complementary fixes close this gap:

Before — Vulnerable
function initialise(address _owner) public {
    require(!initialised, "DeFiTreasury: already initialised");
    initialised = true;
    owner = _owner; // ❌ anyone can be the "first" caller
}
After — Fixed
// ✅ Option 1: use OpenZeppelin's Initializable + deploy-and-init in one atomic transaction
// via a factory contract, removing the public window entirely
import "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";

contract DeFiTreasury is Initializable {
    address public owner;

    // ✅ Deployer address pinned at factory-deploy time, compared on call
    function initialise(address _owner, address _expectedDeployer) public initializer {
        require(msg.sender == _expectedDeployer, "DeFiTreasury: unauthorised initialiser");
        owner = _owner;
    }
}
Production Note The most robust fix is architectural: deploy and initialise in the same transaction using a factory contract, so there is never a block in which the proxy exists in an uninitialised, publicly callable state. OpenZeppelin's Initializable modifier (paired with this factory pattern) is the industry-standard approach and is directly relevant here, as it was already flagged as a reference pattern earlier in this training programme.
