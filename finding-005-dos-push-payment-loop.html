Security Audit Finding #005

Permanent Denial of Service via
Push-Payment Refund Loop

InvestmentPool.sol Custom training example Educational Audit Exercise
High
The contract refunds investors by iterating an array and pushing ETH to each address. A single participant using a contract address that rejects incoming ETH causes the entire loop to revert on every attempt — permanently freezing all investor funds. No attacker profit is possible, but every legitimate depositor loses access to their principal irrecoverably.
Description

InvestmentPool maintains an array of investor addresses and refunds them via a single refundAll() function that loops over that array, pushing each investor's deposit to their address using transfer().

Solidity transactions are atomic: if any single statement reverts, the entire transaction unwinds — including every operation that had already succeeded earlier in the same call. A loop that performs external transfers therefore has as many points of failure as it has entries.

An attacker exploits this without any privileged access and without ever calling the vulnerable function themselves. They simply participate as an ordinary investor using a contract address engineered to reject ETH — one with no receive() or fallback() function, or one whose fallback deliberately reverts. From that moment, their address sits in the array as a permanent landmine.

Every subsequent call to refundAll() proceeds normally until it reaches the malicious address, at which point the transfer() fails, the revert propagates, and the whole refund — for every investor — is rolled back. The attacker is entirely passive; the contract breaks itself against their address.

Vulnerable Code
InvestmentPool.sol — refundAll() (vulnerable)
contract InvestmentPool {
    address[] public investors;
    mapping(address => uint256) public deposits;

    function deposit() external payable {
        if (deposits[msg.sender] == 0) investors.push(msg.sender);
        deposits[msg.sender] += msg.value;
    }

    // ❌ Loops over external addresses performing transfers
    // ❌ One rejecting address reverts the entire call, for everyone
    function refundAll() external onlyOwner {
        for (uint i = 0; i < investors.length; i++) {
            payable(investors[i]).transfer(deposits[investors[i]]);
            deposits[investors[i]] = 0;
        }
    }
}

// The attacker's participating contract — note the absence of receive()/fallback()
contract RejectingInvestor {
    function joinPool(address pool) external payable {
        InvestmentPool(pool).deposit{value: msg.value}();
    }
    // ❌ No receive() and no fallback() — all incoming ETH reverts
}
Impact

Once a single rejecting address is present in the investor array:

  • refundAll() reverts on every call and can never complete
  • All investor principal remains locked in the contract with no alternative withdrawal path
  • The freeze is permanent — the contract is immutable and the malicious entry cannot be removed
  • Investors who joined in good faith lose 100% of their deposited funds
  • The attacker gains nothing and also loses their own deposit — this is a pure griefing attack
Real-World Precedent — GovernMental (2016) The GovernMental Ponzi contract accumulated a creditor list so large that paying it out exceeded the block gas limit, permanently stranding roughly 1,100 ETH. Different trigger — unbounded array growth rather than a rejecting recipient — but the same structural flaw: a single loop over external payouts became an all-or-nothing operation that could no longer complete, freezing every participant's funds at once.
Severity Rationale — Why High, Not Critical No funds are stolen and the attacker derives no profit — indeed they sacrifice their own deposit to execute it. That absence of attacker gain is why this is not rated Critical. However, the loss to legitimate users is total and irreversible: every depositor's principal is permanently unrecoverable, and the protocol's core function is permanently disabled. Permanent, protocol-wide fund freezing places this firmly at High.
Proof of Concept

Requires no privileged access. The attacker never calls the vulnerable function.

  1. Attacker deploys RejectingInvestor — a contract with no receive() or fallback() function, so any incoming ETH transfer reverts.
  2. Attacker deposits into the pool through that contract:
    RejectingInvestor.joinPool{value: 1 ether}(poolAddress)
    Their contract address is now appended to investors[].
  3. Legitimate investors continue depositing normally, unaware of the planted entry.
  4. The owner eventually calls refundAll() to return funds.
  5. The loop refunds investors successfully until it reaches the attacker's address, where transfer() reverts.
  6. Atomicity rolls back the entire transaction — including all refunds already paid within that call. No investor receives anything.
  7. Every retry produces the identical result. With no alternative withdrawal function, all deposits are frozen permanently.
Cost to attacker: deployment gas + their own forfeited deposit  |  Attacker profit: none — pure griefing
Recommendation

Replace the push-payment loop with a pull-payment pattern: rather than the contract distributing funds to every investor in one transaction, each investor withdraws their own balance individually. This removes the shared point of failure entirely — a rejecting contract can then only ever break its own withdrawal.

Before — Vulnerable (push)
function refundAll() external onlyOwner {
    for (uint i = 0; i < investors.length; i++) {
        payable(investors[i]).transfer(deposits[investors[i]]); // ❌ shared failure point
        deposits[investors[i]] = 0;
    }
}
After — Fixed (pull)
// ✅ Each investor withdraws independently — no loop, no shared failure point
// ✅ Checks-Effects-Interactions: state cleared before the external call
function withdrawRefund() external {
    uint256 amount = deposits[msg.sender];
    require(amount > 0, "Nothing to withdraw");

    deposits[msg.sender] = 0;

    (bool success, ) = payable(msg.sender).call{value: amount}("");
    require(success, "Transfer failed");
}
General Principle Never loop over an array of external addresses performing transfers or calls. Any single uncooperative entry holds the entire operation hostage, and on an immutable contract that hostage situation is permanent. Let recipients pull their own funds. Note also the switch from transfer() to call{value:} in the fix: transfer()'s fixed 2300-gas stipend can fail against legitimate smart-contract wallets, so call with an explicit success check — combined with state updated beforehand — is the safer modern pattern.
